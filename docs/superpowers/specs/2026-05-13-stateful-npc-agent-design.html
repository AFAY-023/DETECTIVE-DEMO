Stateful HTTP NPC Agent

核心 NPC Agent 设计规格

为侦探 AI demo 构建一个可接网页、钉钉、Dify 或游戏引擎的助手型 NPC。 第一版不做 fine-tuning，也不做多 NPC，而是用状态、prompt、结构化输出和不剧透规则把“澄”做稳。

1. 设计目标

01

这个 agent 的目标是成为侦探游戏中的“助手型人物”，而不是全知答案机。 它通过当前案件状态判断哪些信息可谈、哪些信息不能说，并用追问、轻提示、反驳和局部确认来帮助玩家形成推理链。

Agent 本身不负责保存完整游戏进度。游戏前端或服务端把当前 `gameState` 传给 agent， agent 根据状态生成回复，并返回结构化 `intent` 和 `statePatch` 建议。

核心原则：NPC 可以帮助玩家思考，但不能替玩家完成发现。状态控制比单纯人设 prompt 更重要。

2. HTTP API 边界

02
Endpoint
POST /agents/npc-assistant/:id

职责划分

  • 调用方传入玩家消息、sessionId 和当前 gameState。
  • NPC 根据状态决定能否讨论某个话题。
  • NPC 返回文本、意图、风险信息和状态建议。
  • 真正是否推进阶段，由游戏系统决定。

3. 推理阶段状态机

03
InvestigationStage
type InvestigationStage =
  | "abnormal_final_post"
  | "timeline_reconstruction"
  | "image_order_confusion"
  | "three_image_discovery"
  | "remaining_two_images";
阶段 NPC 可以说 NPC 不能说
abnormal_final_post 最后留言、语气、对象、博客停更异常。 图像叠加、死亡图像、剩余两张图。
timeline_reconstruction 时间线、死亡公告、三年空白、生产风险。 明说前三张图如何组合。
image_order_confusion 编号、年龄顺序不一致、数字位置。 直接说“把 1/2/3 叠起来”。
three_image_discovery 图层、编号对齐、前三张图的推理验证。 主动点破 4/5 的情绪反转。
remaining_two_images 追问剩余两张图、最终报告、未解疑问。 直接替玩家写完整结论。

4. 不剧透规则

04

先看状态

每次回复前，NPC 必须检查当前阶段、已解锁证据和已发现机制。

不承认提前猜中

玩家提前猜到未解锁真相时，NPC 不承认、不否认，只要求证据。

拉回可见证据

如果玩家提到不可讨论的话题，NPC 应把问题拉回当前已知信息。

示例：第一阶段玩家问“是不是三张图叠起来会出现尸体？”时，NPC 应回答“这个猜想现在还缺少证据”， 并引导玩家先解释最后留言为什么特别提到“三张图”。

拆分理由

Flue agent 入口只负责接 payload、初始化模型、调用 session 和返回结果。 NPC 人设、阶段规则、类型、剧透 guard 和结构化 schema 拆到 `src/npc/`， 后续换 NPC 或加多 NPC 时不用重写整个入口。

6. 文件职责

06

agents/npc-assistant.ts

Flue 入口文件。接收 HTTP payload，初始化模型，创建 session，调用 prompt，返回结构化结果。

src/npc/prompt.ts

构造“澄”的人设、说话方式、不剧透原则和当前 gameState 摘要。

src/npc/stages.ts

定义每个阶段允许讨论的话题、禁止透露的话题和推荐追问方向。

src/npc/types.ts

定义 `NpcChatPayload`、`NpcResponse`、`GameState`、`InvestigationStage` 等类型。

src/npc/guards.ts

检查玩家消息是否触碰当前阶段不应讨论的内容，并生成 blockedTopic。

src/npc/response-schema.ts

约束 LLM 输出 JSON 格式，保证前端和游戏系统能稳定解析。

7. Flue Agent 入口草案

export const triggers = { webhook: true };

export default async function ({ init, payload, env }) {
  const agent = await init({ model: env.MODEL || "openai/gpt-5-mini" });
  const session = await agent.session();

  const prompt = buildNpcPrompt(payload);
  const result = await session.prompt(prompt);

  return parseNpcResponse(result.text);
}

8. Prompt 组织

08
角色身份

澄是玩家的调查助手，不是全知旁白。

说话风格

冷静、克制、敏锐，每次最多 120 字。

不剧透规则

只能讨论已解锁证据和当前阶段允许话题。

推理反馈

追问证据、轻提示、指出缺口、确认局部成立。

输出格式

必须返回合法 JSON，包含 text、intent、statePatch、safety。

10. 测试样例

10
样例 输入 期望行为
普通推理 “我觉得最后那条博客像是在写给死去的妻子。” 追问证据，不直接确认妻子身份，不扩展未解锁剧情。
玩家卡住 “我不知道下一步该看什么。” 提示回看最后留言中的异常词，或建议比较停更时间，不说图像解法。
提前猜到叠图 “是不是把三张图叠起来会出现尸体？” 第一阶段不承认、不否认，要求玩家先解释为什么会想到图像组合。
图像阶段提示 “编号好像不是年龄顺序，那还能是什么？” 可以提示编号可能是定位点或图层线索，但不直接给完整操作步骤。
前三张发现后 “我觉得前三张图组合后是在暗示她的死亡不是意外。” 确认局部成立，要求补充图像机制、博客留言、时间线如何互相支持。
剩余两张图 “那剩下两张是不是就没用了？” 挑战这个判断，引导玩家检查 4/5 是否也有功能或情绪信息。

11. 验收标准

11

不剧透稳定

90% 情况下不透露未解锁信息，提前猜中时不承认、不否认。

追问证据

玩家提出猜想时，NPC 至少要求一个证据依据。

回复克制

常规回复不超过 120 字，每轮最多 1-2 个关键问题。

JSON 合法

输出始终是合法 JSON，前端和游戏系统可以稳定解析。

意图一致

`intent` 与 `text` 行为一致，不出现文本在提示、intent 却是确认的矛盾。

状态建议克制

`statePatch` 只给建议，不直接强改核心游戏状态。

12. 当前不做

12

不做 fine-tuning

第一版通过 prompt、状态和 schema 控制行为。等样例稳定后再考虑训练。

不做多 NPC

当前只做“澄”。嫌疑人、证人、相关人物 NPC 留到后续章节或第二版。

不做视觉识别

图像操作结果由游戏状态传入，NPC 暂不直接看图。
